# Offline Screen Monitor Requirements

## 1. Purpose

폐쇄망 환경에서 전송측 Windows PC 또는 Windows Server의 현재 화면을 수신측 일반 Windows PC에서 지속적으로 모니터링한다.

상용 원격제어 프로그램을 사용하지 않고, 내부용으로 직접 만든 최소 기능 앱을 전송측과 수신측에 각각 배포한다.

이 문서는 PoC를 대상으로 한다. 현재는 전송측 1대, 수신측 1대의 1:1 구성이지만, 성공하면 유휴 장비를 전송측으로 추가 투입할 가능성이 있다. 프로토콜은 그 확장을 미리 수용한다.

## 2. Terms

- 전송측(Sender): 화면을 캡처해서 보내는 Windows PC 또는 Windows Server. 물리적으로 멀리 있고, 접근 비용이 높다
- 수신측(Receiver): 전송측 화면을 표시하는 일반 Windows PC. 관리자가 언제든 직접 조작할 수 있다
- 화면 스트림: 전송측이 반복 캡처한 화면 이미지를 네트워크로 전송하는 데이터 흐름
- 제어 신호: 수신측이 전송측에게 보내는 캡처 일시정지/재개 신호. 원격 제어가 아니다

## 3. Operating Environment

- 전송측 OS: Windows 또는 Windows Server
- 수신측 OS: 일반 Windows
- 네트워크: 완전 오프라인 폐쇄망, 인터넷 연결 없음
- 전송측과 수신측 간 직접 TCP 통신 가능
- 오디오 전송: 불필요
- 원격 제어: 불필요
- 인증/암호화: 불필요
- 상용 프로그램 또는 상용 라이선스 의존성: 사용 불가

## 4. Core Constraints

이 프로젝트의 모든 설계 결정은 아래 비대칭성에서 나온다.

**전송측은 건드릴 수 없다.**

- 최초 1회 설치 후 사실상 손대지 않는다
- 오버헤드로 죽으면 안 된다. 유휴 시 CPU 점유가 무시할 수준이어야 한다
- 간혹 재부팅되며, 이때 사람 개입 없이 알아서 올라와야 한다
- 모든 설정은 빌드 또는 설치 시점에 확정한다. 런타임 설정 UI도, 설정 파일 수정도 없다
- 전송측 IP는 간혹 바뀐다. 전송측 주소에 의존하는 설계는 금지한다

**수신측은 마음껏 건드릴 수 있다.**

- 관리자가 직접 조작 가능한 위치에 있다
- 고정 IP이며 거의 절대 변하지 않는다. 변한다면 전송측을 새로 빌드해서 재설치한다
- 방화벽 설정, 재시작, 업데이트 부담을 전부 이쪽으로 몰아도 된다

즉 복잡도는 전부 수신측으로 밀어낸다.

## 5. Network Direction

**전송측이 수신측으로 접속을 거는 push 방식으로 한다.**

```text
전송측 -> 수신측 (TCP connect)
전송측 -> 수신측 (JPEG 프레임)
전송측 <- 수신측 (일시정지/재개 신호만)
```

근거:

- TCP 연결을 받는 쪽(listen)은 안정적인 주소와 inbound 방화벽 규칙이 필요하다. 전송측 IP가 바뀌므로 전송측은 listen할 수 없다
- 연결을 거는 쪽(connect)은 자기 IP가 무엇이든 무관하고, outbound는 대개 방화벽 기본 허용이다
- 따라서 inbound 방화벽 규칙은 **수신측에만** 필요하다. 수신측은 관리자가 직접 조작 가능하므로 부담이 없다
- 전송측 설정은 "수신측 IP:PORT 하나"로 끝난다. 빌드에 박아 넣는다

동작:

- 전송측은 부팅 직후부터 수신측으로 접속을 시도한다
- 수신측이 꺼져 있거나 앱이 안 떠 있으면 접속이 실패한다. 전송측은 이를 오류로 취급하지 않고, 재시도 간격을 두고 무한히 재시도한다. 종료하지 않는다
- 접속에 성공하면 프레임 전송을 시작한다
- 연결이 끊기면 즉시 재시도 루프로 돌아간다
- 수신측은 동시에 하나의 전송측 연결만 유지한다. 새 연결이 들어오면 기존 연결을 끊고 교체한다 (§13.3 참조)

## 6. Configuration Policy

설정 화면은 만들지 않는다.

### 6.1 전송측

모든 값을 **빌드 시점에 고정**한다. 커맨드라인 인자도, 설정 파일도, 레지스트리도 읽지 않는다.

빌드 파라미터:

| 항목 | MVP 기본값 |
| --- | --- |
| 수신측 IP | 빌드 시 지정 (필수) |
| 수신측 포트 | 45871 |
| 장비 ID | 빌드 시 지정 (§7.2) |
| FPS | 3 |
| JPEG 품질 | 60 |
| 화면 축소 비율 | 1.0 |
| 재접속 간격 | 5초 (§13.5) |

수신측 IP가 바뀌면 전송측을 새로 빌드해서 재설치한다. 이것이 유일한 갱신 경로다.

### 6.2 수신측

- 대기 포트만 빌드에 고정한다 (전송측과 동일 값)
- 그 외 표시 관련 옵션은 자유롭게 추가해도 된다. 수신측 복잡도는 제약이 아니다

## 7. Protocol

TCP, little-endian.

### 7.1 연결 수립

전송측이 접속에 성공하면 **핸드셰이크 1회**를 먼저 보낸다.

```text
[MAGIC       4 bytes]  "OSM\x01"
[VERSION     1 byte ]  프로토콜 버전. MVP는 0x01
[DEVICE_ID  16 bytes]  UTF-8, 우측 0x00 패딩
```

수신측은 MAGIC이 다르면 즉시 연결을 끊는다. VERSION이 자신이 아는 값보다 크면 연결을 끊고 사용자에게 알린다.

### 7.2 DEVICE_ID

MVP는 1:1이므로 수신측은 이 값을 표시만 하고 분기에 쓰지 않는다.

그럼에도 **지금 프로토콜에 넣는다.** 나중에 유휴 장비를 전송측으로 추가하면 수신측이 여러 연결을 받아 구분해야 하는데, 그때 프로토콜을 바꾸면 이미 배포된 원격지 전송측을 전부 다시 만져야 한다. 전송측은 한 번 심으면 못 만진다고 가정한다. VERSION 바이트도 같은 이유다.

### 7.3 전송측 -> 수신측 메시지

```text
[TYPE  1 byte]
[payload ...]
```

| TYPE | 이름 | payload |
| --- | --- | --- |
| 0x01 | FRAME | `[len 4B][width 4B][height 4B][jpeg bytes]` (len은 jpeg 바이트 수) |
| 0x02 | PING | 없음 |

프레임마다 width/height를 실어 보낸다. 전송측 해상도나 모니터 구성이 런타임에 바뀌어도 재연결 없이 수신측이 따라간다.

PING은 일시정지 상태에서 죽은 연결을 감지하기 위한 것이다 (§13.3).

### 7.4 수신측 -> 전송측 메시지

```text
[TYPE 1 byte]
```

| TYPE | 이름 | 의미 |
| --- | --- | --- |
| 0x10 | PAUSE | 캡처를 중단하라. 연결은 유지한다 |
| 0x11 | RESUME | 캡처를 재개하라 |

전송측은 이 두 값 외의 바이트를 **조용히 무시**한다. 알 수 없는 TYPE을 받았다고 연결을 끊거나 종료하지 않는다. 이것이 전송측을 안 건드리고 수신측만 개선할 수 있게 하는 장치다.

이 역방향 채널은 **화면 캡처의 시작/정지만** 제어한다. 키보드, 마우스, 명령 실행 등 어떤 형태의 원격 제어도 프로토콜에 존재하지 않는다.

## 8. Functional Requirements

### 8.1 전송측

- 실행되면 즉시 수신측 IP:PORT로 TCP 접속을 시도한다
- 접속 실패 시 §13.5의 간격으로 무한 재시도한다. 어떤 경우에도 프로세스가 종료되지 않는다
- 접속 성공 시 핸드셰이크를 보내고 프레임 전송을 시작한다
- 현재 로그인된 사용자 세션의 전체 가상 화면(Virtual Screen)을 캡처한다
- 멀티 모니터가 있으면 모든 모니터를 하나의 큰 이미지로 합쳐서 전송한다
- 캡처한 화면을 JPEG로 압축해서 지정 FPS로 전송한다
- PAUSE를 받으면 캡처 루프를 멈춘다. 이때 CPU 점유는 0에 수렴해야 한다
- RESUME을 받으면 캡처를 재개한다
- 연결이 끊기면 캡처를 중단하고 재접속 루프로 돌아간다
- UI는 없거나 트레이 아이콘만 둔다
- 키보드, 마우스, 클립보드, 파일 전송, 오디오 기능은 제공하지 않는다
- Windows 재부팅 후 사람 개입 없이 자동 시작되어야 한다 (§9)

### 8.2 수신측

- 실행 시 지정 포트에서 전송측 연결을 대기한다
- 핸드셰이크를 검증하고 DEVICE_ID를 표시한다
- 수신한 JPEG 프레임을 창에 표시한다
- 창 크기에 맞게 화면을 축소/확대한다
- 창이 최소화되었거나 화면이 보이지 않는 상태이면 PAUSE를 보낸다. 다시 보이면 RESUME을 보낸다
- 연결이 끊기면 다시 대기 상태로 돌아가고, 그 사실을 화면에 표시한다
- 전체 화면 표시 기능은 있으면 좋지만 MVP 필수는 아니다

## 9. Startup Requirements

전송측은 재부팅 후 사람 개입 없이 올라와야 한다. 원격지라 잠금 화면에서 멈추면 아무도 풀어줄 수 없다.

**전송측 PC에 자동 로그인(auto-login)을 설정하는 것을 전제로 한다.** 이것이 없으면 캡처할 데스크톱 자체가 존재하지 않는다.

방식:

1. 자동 로그인 활성화
2. 작업 스케줄러에 "로그온 시 실행" 트리거로 전송측 앱 등록
3. 설치 스크립트가 위 두 가지를 함께 수행한다

Windows 서비스로 만들지 않는다. 서비스는 Session 0에서 실행되며, Session 0에는 캡처할 화면이 없다. 서비스로 만들려면 활성 세션에 별도 프로세스를 주입해야 하는데(`WTSGetActiveConsoleSessionId` + `CreateProcessAsUser`), 자동 로그인이 가능한 이상 그 복잡도를 감수할 이유가 없다.

전송측에서 화면 보호기와 절전 모드의 **잠금 기능을 해제**해야 한다. 디스플레이 전원이 꺼지는 것 자체는 캡처에 지장이 없지만, 잠금 화면으로 전환되면 검은 화면만 캡처된다.

## 10. Performance Targets

목표는 실시간 원격 조작이 아니라 **가끔 들여다보는 모니터링**이다.

- FPS: 3 (범위 1~5)
- JPEG 품질: 60 (범위 50~70)
- 화면 축소 비율: 1.0 (범위 0.5~1.0)
- 유휴 시(PAUSE 상태) 전송측 CPU 점유: 0에 수렴

이 정도 프레임 간격이면 GDI `BitBlt` 기반 캡처로도 전송측 부하가 충분히 낮다. DXGI Desktop Duplication은 MVP 범위 밖이다.

전송측은 프레임을 큐에 쌓지 않는다. 소켓 전송이 밀리면 오래된 프레임을 버리고 최신 프레임만 보낸다. 네트워크가 느려져도 지연이 누적되지 않아야 한다.

## 11. Technical Approach

- 언어/프레임워크: C# / .NET 8 (LTS)
- 배포 형태: self-contained single-file publish. 대상 PC에 .NET 런타임 설치가 필요 없다
- 전송측 UI: 없음 또는 트레이 아이콘
- 수신측 UI: WinForms
- 화면 캡처: `Graphics.CopyFromScreen` (GDI BitBlt)
- 멀티 모니터 좌표: `SystemInformation.VirtualScreen`. 보조 모니터가 주 모니터 왼쪽/위쪽에 있어 좌표가 음수인 경우도 처리한다
- 이미지 압축: JPEG
- 통신: TCP

빌드에는 .NET SDK가 필요하다. 현재 개발 PC에는 런타임만 설치되어 있다.

## 12. Security Requirements

폐쇄망 내부 전용이며, 인증과 암호화는 구현하지 않는다.

- 사전 공유 인증 문자열 없음
- 암호화 없음
- 수신측은 접속한 전송측을 검증하지 않는다. MAGIC/VERSION 확인은 프로토콜 오식별 방지용이지 인증이 아니다

수신측이 전송측에게 보낼 수 있는 것은 PAUSE와 RESUME 두 바이트뿐이다. 전송측은 그 외 어떤 입력도 해석하지 않으므로, 수신측이 침해되어도 전송측을 조작할 수 없다.

## 13. Open Questions / Decisions Needed

아래는 아직 확정되지 않았거나, 구현 전에 확인이 필요한 항목이다.

### 13.1 방화벽 — 해결됨

Windows 방화벽 기본 정책은 outbound 전부 허용, inbound 전부 차단이다. 따라서:

- **전송측은 connect만 하므로 방화벽 규칙도, 사용자 프롬프트도 필요 없다.** 설치 스크립트에서 할 일이 없다
- **수신측은 listen하므로 최초 실행 시 Windows 방화벽 허용 프롬프트가 뜬다.** 관리자가 직접 "허용"을 누르면 끝난다

단, 폐쇄망에 별도 하드웨어 방화벽이나 그룹 정책으로 outbound가 차단된 경우는 Windows 방화벽과 무관하며 프롬프트도 뜨지 않는다. 이 경우 연결이 조용히 실패한다. 최초 연결 시험으로 즉시 판별된다.

### 13.2 전송측 로그 — 결정됨

`%LOCALAPPDATA%\ScreenSender\sender.log`에 남긴다. 1MB를 넘으면 `sender.log.1`로 밀어내고, 백업은 1개만 유지한다. 최대 2MB로 고정된다.

로그 기록 실패(디스크 가득, 권한 등)는 무시한다. 로그 한 줄을 잃는 것이 전송측이 죽는 것보다 항상 낫다.

### 13.3 좀비 연결

전송측이 전원 차단이나 커널 패닉으로 죽으면 TCP FIN이 가지 않는다. 수신측은 연결이 살아 있다고 착각한 채 영원히 기다리고, 전송측이 재부팅 후 새로 접속해도 수신측은 "이미 연결이 있다"고 판단할 수 있다.

대응 방향(결정 필요):

- 전송측이 PAUSE 상태에서도 일정 주기로 PING을 보내고, 수신측은 일정 시간 무소식이면 연결을 끊는다
- 수신측은 새 연결이 들어오면 기존 연결을 무조건 끊고 교체한다 (§5)
- TCP keepalive를 켤지 여부

**결정: PING 30초, 수신측 read 타임아웃 90초.** TCP keepalive는 쓰지 않는다. 애플리케이션 레벨 PING으로 충분하고, OS 설정에 의존하지 않는 편이 낫다.

수신측은 새 연결을 받으면 기존 연결을 무조건 끊는다. 전송측이 재부팅해서 새로 접속하면 좀비 소켓이 남아 있어도 즉시 교체된다.

### 13.4 프레임 크기와 대역폭 — 해결됨

전송측은 1080p 모니터 2대다. Virtual Screen은 3840x1080이다. 품질 60 JPEG 기준 프레임당 대략 200~500KB, 3fps이면 초당 0.6~1.5MB(5~12Mbps)다.

랜 대역폭이 최소 100Mbps이므로 **축소 비율 1.0을 그대로 쓴다.** 축소는 불필요하다.

### 13.5 재접속 간격 — 결정됨

지수 백오프. 5초에서 시작해 실패할 때마다 2배로 늘리고 60초에서 상한을 둔다. 연결에 성공하면 5초로 초기화한다.

수신측이 몇 시간 꺼져 있어도 최대 60초 안에 다시 붙는다.

### 13.6 전송측 headless 문제 — 해당 없음

전송측에는 1080p 모니터 2대가 항상 물리적으로 연결되어 있다. 더미 HDMI 어댑터나 가상 디스플레이는 필요 없다.

모니터가 3대로 늘어날 가능성은 있으나, 그 시점에는 어차피 현장 방문이 필요하므로 재빌드 후 재설치한다. Virtual Screen 전체를 캡처하는 구현이라 코드 변경 없이 동작할 가능성이 높다.

### 13.7 전송측 앱 설치 위치 — 결정됨

`%LOCALAPPDATA%\ScreenSender\ScreenSender.exe`.

전송측 앱은 로그인 사용자 세션에서 도는 사용자 단위 프로그램이고, 자동 로그인 계정 하나만 쓴다. `%ProgramFiles%`에 두면 쓰기 권한 때문에 갱신이 번거로워질 뿐 얻는 게 없다. 로그도 같은 폴더에 남는다.

### 13.8 마우스 커서 — 미결

`CopyFromScreen`은 마우스 커서를 그리지 않는다. 모니터링 용도상 커서 위치가 보이면 유용할 수 있으나, 커서를 합성하려면 별도 코드가 필요하다. MVP에서는 넣지 않는다. 필요해지면 전송측 재빌드 대상이다.

## 14. Known Limitations

- 잠금 화면, Ctrl+Alt+Del 화면, UAC 보안 데스크톱은 캡처가 안 되거나 검은 화면으로 나온다
- 전송측 사용자가 로그아웃하면 캡처가 중단된다
- 전송측은 물리 콘솔 세션에서 실행하는 것을 전제로 한다. 누군가 전송측에 RDP로 접속하거나, 앱이 RDP 세션에서 실행된 뒤 그 세션이 끊기면 화면이 검게 나온다. 이는 오류가 아니라 비정상 상태의 신호로 간주한다
- 네트워크 대역폭은 해상도, FPS, JPEG 품질, 모니터 수에 비례해 증가한다
- 수신측에 접속할 수 있는 전송측을 제한하지 않는다. 폐쇄망 신뢰를 전제로 한다

## 15. Explicit Non-Goals

- 인증 및 암호화
- 원격 키보드/마우스 제어
- 파일 전송, 클립보드 공유, 오디오, 채팅
- 사용자 계정 관리
- 복잡한 설정 UI
- 인터넷 중계 서버, 클라우드 연동
- 상용 원격제어 솔루션 연동
- 영상 회의 수준의 고품질 스트리밍
- 중계 서버(relay). 수신측이 고정 IP이므로 불필요하다

MVP 범위 밖이지만 프로토콜은 수용하는 것:

- 다중 전송측 동시 접속 (DEVICE_ID로 구분)
- 특정 모니터만 전송, 모니터별 분리 표시
- DXGI Desktop Duplication, H.264 인코딩

## 16. MVP Acceptance Criteria

- 수신측 앱을 실행하면 지정 포트에서 대기한다
- 전송측 앱을 실행하면 수신측에 접속하고, 전송측 화면이 수신측 창에 계속 표시된다
- 수신측 앱이 꺼져 있는 상태에서 전송측을 실행해도 전송측이 종료되지 않는다. 이후 수신측을 켜면 자동으로 연결된다
- 수신측 앱을 껐다 켜면 전송측이 재접속한다. 전송측을 만지지 않는다
- 전송측 멀티 모니터 화면이 하나의 큰 화면으로 표시된다
- 수신측 창 크기를 바꾸면 화면이 맞춰서 축소/확대된다
- 수신측 창을 최소화하면 전송측 CPU 점유가 0에 수렴한다. 복원하면 화면이 다시 흐른다
- 수신측에서 키보드/마우스로 전송측을 제어할 수 없다
- 전송측 PC를 재부팅하면 사람 개입 없이 앱이 자동 시작되어 재접속한다
- 전송측 PC의 IP를 바꿔도 아무 설정 변경 없이 계속 동작한다
